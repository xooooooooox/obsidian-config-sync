# 1.0.0 Documentation Pass Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bring the user-facing README up to the 1.0.0 feature set and add a maintainer-facing `docs/ARCHITECTURE.md`, with no code or behavior changes.

**Architecture:** Pure documentation. `docs/ARCHITECTURE.md` is written first (it's independent), then the README English text is refreshed in place, screenshots are regenerated from the current build in the dev vault, and `README.zh.md` is mirrored. A final pass verifies every claim against the named source files and confirms the gates are still green.

**Tech Stack:** Markdown. obsidian-cli (screenshots, dev vault). vitest / eslint / `scripts/check-no-hardcoded-color.sh` (gate confirmation only — no source is touched).

## Global Constraints

- No code or behavior changes; no new features; no rewrite of already-accurate README sections.
- Bilingual parity: every README change mirrors into `README.zh.md`.
- Privacy (user rule): no real home paths / usernames / hostnames in written artifacts — use `~`, `$HOME`, `<host>`, `${VAR}`.
- No Claude/AI attribution in any commit message.
- Never commit unless the executing user explicitly asks (keep changes uncommitted for review); this plan's "commit" steps run only under that explicit instruction.
- Every ARCHITECTURE.md claim must match the actual code — no invented functions or drifted names.
- Screenshots regenerated from the dev vault (`dev/vault/`, current 1.0.0 build already installed); obsidian-cli routes by CWD, so run screenshot commands from `dev/vault/`.
- Source of truth for module responsibilities: `src/core/*.ts`, `src/ui/*.ts`, `src/main.ts`.
